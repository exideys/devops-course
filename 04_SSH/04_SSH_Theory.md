У Темі 3 ми налаштували мережу і переконалися, що з нашого хост-комп'ютера можна дістатися до VM за адресою `192.168.56.10`. **Зв'язок з проєктом:** Реальний VPS проєкту My AI Assistant доступний виключно через SSH-ключ — пароль для root відключений повністю. CI/CD у GitHub Actions використовує SSH-ключ, збережений як секрет, для деплою на сервер.

## 🎭 Метафора заняття: Ключ і замки

Уявіть, що у вас є **унікальний ключ** — ваш приватний ключ (`id_ed25519`). Ви носите його завжди при собі і нікому не віддаєте.

- **Приватний ключ (`id_ed25519`)** — це ваш **ключ**. Він унікальний. Ви ніколи нікому його не передаєте і не випускаєте з рук.
- **Публічний ключ (`id_ed25519.pub`)** — це **замок, виготовлений спеціально під ваш ключ**. Ви можете замовити таких замків скільки завгодно і встановити їх будь-де: на сервер у Варшаві, у Берліні, на GitHub. Жоден інший ключ ці замки не відкриє — вони зроблені виключно під ваш.
- **`authorized_keys` на сервері** — це місце, де встановлений замок. Коли ви підключаєтесь, сервер просто перевіряє: цей ключ підходить до замка? Якщо так — відкриває. Якщо ні — відмовляє.

Хочете закрити комусь доступ? Просто знімаєте замок із сервера (видаляєте публічний ключ з `authorized_keys`). Ваш приватний ключ при цьому залишається у вас без змін.

Паролю в цій схемі **немає місця**. Ні людина, ні скрипт не вводять нічого — тільки пред'являють ключ.

## 🧠 Необхідні знання

### 1. Асиметрична криптографія

Звичайна криптографія (симетрична) використовує один і той самий ключ для шифрування й розшифрування — як замок і ключ у побуті. Проблема: цей ключ треба якось передати іншій стороні, але по яким каналом? Якщо канал ненадійний — ключ можна перехопити.

Асиметрична криптографія вирішує цю проблему елегантно: використовуються **два математично пов'язані ключі**.

- **Публічний ключ** — можна вільно роздавати. Він шифрує (або перевіряє).
- **Приватний ключ** — зберігається тільки у вас. Він розшифровує (або підписує).

Математичний зв'язок між ними побудований на **односторонніх функціях** — операціях, які легко виконати в одному напрямку, але практично неможливо обернути. Наприклад: перемножити два великих прості числа займає мілісекунди, а розкласти результат назад на множники — мільйони років (при достатній довжині ключа).

Це означає: навіть якщо зловмисник отримає ваш публічний ключ — він не зможе відновити приватний. SSH будує свою автентифікацію саме на цьому принципі.

**Ключові поняття:**

- **Публічний ключ** — розповсюджується вільно. Знищення безпеки не несе.
- **Приватний ключ** — захищається як пароль. Його компрометація = компрометація всіх серверів, де є ваш публічний ключ.
- **Одностороння функція** — математична основа асиметричної криптографії.

### 2. Протокол SSH: як відбувається з'єднання

SSH (Secure Shell) — це не просто «захищений термінал». Це повноцінний криптографічний протокол, який вирішує два завдання:

1. **Ви впевнені, що підключаєтесь до правильного сервера** (а не до зловмисника посередині).
2. **Сервер впевнений, що ви — це ви** (автентифікація по ключу).

#### 2.1. Перевірка особистості сервера: `known_hosts`

Коли ви вперше підключаєтесь до сервера, SSH показує щось схоже на:

```text
The authenticity of host '192.168.56.10 (192.168.56.10)' can't be established.
ED25519 key fingerprint is SHA256:wJ8+7aP1Kxq3v9Lmz2nRsT0uYcOdEfGhIjKlMnOpQr=
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

Це НЕ помилка. SSH питає: «Ти вперше бачиш цей сервер — довіряти йому?». Якщо відповісти `yes`, SSH збереже «відбиток» сервера у файл `~/.ssh/known_hosts`. При наступних підключеннях він буде порівнювати відбиток — і якщо сервер раптом видасть інший, підніме тривогу. Саме так SSH захищає від атаки «людина посередині» (MITM).

> ⚠️ **Якщо SSH повідомляє `WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!`** — це серйозний сигнал. Або сервер перевстановили, або між вами і сервером хтось намагається перехопити з'єднання. Перш ніж ігнорувати — розберіться.

#### 2.2. Автентифікація по ключу: як замок перевіряє ключ

Тут виникає природне питання: якщо є ключ і замок — чому б просто не «вставити» ключ через мережу? Відповідь проста: **надсилати приватний ключ через мережу небезпечно**. Хтось може перехопити його в дорозі — і тоді замок стане марним.

Тому SSH використовує хитріший підхід: замок **ніколи не просить надіслати ключ**. Натомість він переконується, що ключ у вас є, через маленьку математичну головоломку:

1. **Замок надсилає головоломку** — випадковий набір даних, зашифрований вашим публічним ключем (замком). Розшифрувати його може лише той, у кого є відповідний приватний ключ.
2. **Ваш SSH-клієнт автоматично вирішує головоломку** приватним ключем і надсилає відповідь.
3. **Замок перевіряє відповідь.** Правильна — значить ключ є. Відкрити.

Ваш приватний ключ **ніколи не покидає ваш комп'ютер**. Через мережу летить лише «доказ» того, що він у вас є. Це і є сенс асиметричної криптографії.

```text
 [Ваш комп'ютер]                            [Сервер (VM)]
        │                                         │
        │─── 1. "Хочу підключитись як vagrant" ──►│
        │                                         │
        │◄── 2. "Добре. Ось головоломка —         │
        │        вирішити її може лише той,       │
        │        у кого є твій ключ"──────────────│
        │                                         │
        │  [SSH автоматично вирішує               │
        │   головоломку приватним ключем]         │
        │                                         │
        │─── 3. "Ось відповідь" ─────────────────►│
        │                                         │
        │          [перевіряє відповідь:          │
        │           тільки власник ключа          │
        │           міг її дати]                  │
        │                                         │
        │◄── 4. Відповідь вірна. Ласкаво просимо! │
```

Сервер ніколи не отримує приватний ключ. Він надсилає задачу і перевіряє результат. Без справжнього ключа — задачу вирішити неможливо.

**Ключові поняття:**

- **`known_hosts`** — локальний список довірених серверів (їхніх відбитків). Захист від MITM.
- **Challenge-response** — механізм автентифікації SSH: сервер надсилає головоломку (challenge), клієнт вирішує її приватним ключем і повертає відповідь (response).
- **MITM (Man-in-the-Middle)** — атака перехоплення, коли зловмисник стоїть між клієнтом і сервером.

### 3. Генерація SSH-ключів: `ssh-keygen`

Ключі генеруються на **хост-машині** (вашому ноутбуці/ПК) — там, звідки ви підключаєтесь. Ніколи не генеруйте ключі на сервері і не копіюйте туди приватний ключ.

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

- `-t ed25519` — алгоритм **Ed25519**. Сучасний стандарт: короткі ключі, швидкий, математично надійний. Якщо стикаєтесь зі старим обладнанням, яке не підтримує Ed25519 — використовуйте RSA 4096 (`-t rsa -b 4096`).
- `-C "..."` — коментар. Допомагає пізніше зрозуміти, для чого цей ключ. Технічно ніяк не впливає на безпеку.

Команда запитає два питання:

**Куди зберегти ключ?**

```text
Enter file in which to save the key (/home/user/.ssh/id_ed25519):
```

Натисніть Enter — стандартне місце підходить. За замовчуванням SSH автоматично їх знаходить.

**Пароль для ключа (passphrase)?**

```text
Enter passphrase (empty for no passphrase):
```

Passphrase — це додатковий рівень захисту. Навіть якщо хтось вкраде файл вашого приватного ключа — без passphrase він марний. Для навчального середовища — можна залишити порожнім (Enter). Для Production — обов'язково встановіть.

В результаті у `~/.ssh/` з'являться два файли:

| Файл           | Що це                                                        | Куди відправляти       |
| ------------------ | ---------------------------------------------------------------- | ------------------------------------- |
| `id_ed25519`     | **Приватний ключ** — ваш секрет     | Нікуди. Тільки у вас. |
| `id_ed25519.pub` | **Публічний ключ** — ваш відбиток | На сервер, в GitHub тощо |

**Ключові поняття:**

- **Ed25519** — рекомендований алгоритм для нових ключів (2024+).
- **Passphrase** — пароль для захисту файлу приватного ключа. Не плутати з паролем сервера.
- **`~/.ssh/`** — стандартна директорія для ключів і конфігурацій SSH. Має права `700` (тільки власник).

### 4. Копіювання ключа на сервер: `ssh-copy-id` і `authorized_keys`

Публічний ключ треба покласти на сервер так, щоб SSH-демон його знайшов. Стандартне місце — файл `~/.ssh/authorized_keys` у домашній директорії користувача, під яким ви будете підключатись.

#### Що Vagrant уже зробив за нас

Якщо ви підключались командою `vagrant ssh` — ви вже користувались SSH-ключами, просто не помічали цього. При запуску VM Vagrant автоматично:

1. Генерує пару SSH-ключів
2. Кладе публічний ключ у `~/.ssh/authorized_keys` користувача `vagrant`
3. Зберігає приватний ключ локально і підставляє його при кожному `vagrant ssh`

Саме тому `vagrant ssh` ніколи не питає пароль — це і є безпарольна автентифікація по ключу. Ми будемо робити те саме для користувача `openclaw`, але вручну — щоб зрозуміти кожен крок.

#### Реальний сценарій: налаштовуємо доступ для `openclaw`

У Темі 2 ми створили системного користувача `openclaw`. Нам потрібно мати можливість підключатись до VM безпосередньо як `openclaw` — для адміністрування, налагодження та CI/CD. Vagrant тут не допоможе: він керує тільки своїм користувачем.

Але в нас вже є привілейований канал — підключення через `vagrant`, який має права `sudo`. Саме через нього ми і «підкладемо» ключ.

Це класичний DevOps-патерн: **bootstrap через вже існуючий довірений канал**. На реальному VPS у ролі `vagrant` виступає `root` або інший адміністративний користувач, доступ до якого провайдер надає при першому підключенні.

#### Підготовка: даємо `openclaw` можливість увійти

У Темі 2 ми створили `openclaw` як системного користувача з прапором `--system`. Це автоматично виставляє shell `/usr/sbin/nologin` — спеціальна програма, яка одразу завершує сесію з повідомленням «This account is currently not available».

Перевіримо, що зараз встановлено:

```bash
vagrant ssh
getent passwd openclaw
```

Вивід буде схожий на:

```text
openclaw:x:998:998:OpenClaw AI Agent:/home/openclaw:/usr/sbin/nologin
```

Останнє поле — це shell. `/usr/sbin/nologin` — заборона інтерактивного входу.

> 💡 **Чому так зроблено в Темі 2?** Це принцип найменших привілеїв: сервісний процес не повинен мати можливість «розмовляти» з системою через термінал. Якщо агент скомпрометований — зловмисник не отримає shell. Але для нас, як адміністраторів, SSH-доступ потрібен — для налагодження та роботи з VS Code Remote. Безпека при цьому не порушується: `openclaw` все одно не має `sudo`.

Змінимо shell на `/bin/bash`:

```bash
# Все ще на VM, від імені vagrant:
sudo chsh -s /bin/bash openclaw

# Перевіримо:
getent passwd openclaw
# openclaw:x:998:998:OpenClaw AI Agent:/home/openclaw:/bin/bash
```

Тепер `openclaw` може мати інтерактивну сесію — але без жодних адміністративних прав.

**Ключове поняття:**

- **`chsh`** (change shell) — команда для зміни login shell користувача.
- **`/usr/sbin/nologin`** — спеціальний «shell», який забороняє інтерактивний вхід. Стандарт для сервісних облікових записів.

**Крок 1.** Підключаємось до VM через `vagrant` і кладемо свій публічний ключ в `authorized_keys` користувача `openclaw`:

```bash
# Прочитати свій публічний ключ
cat ~/.ssh/id_ed25519.pub
```

Скопіюйте вивід. Потім підключіться до VM і виконайте:

```bash
vagrant ssh

# На VM — від імені vagrant (через sudo):
sudo mkdir -p /home/openclaw/.ssh
echo "ВСТАВТЕ_ВАШ_ПУБЛІЧНИЙ_КЛЮЧ" | sudo tee /home/openclaw/.ssh/authorized_keys

# Встановити правильні права та власника
sudo chmod 700 /home/openclaw/.ssh
sudo chmod 600 /home/openclaw/.ssh/authorized_keys
sudo chown -R openclaw:openclaw /home/openclaw/.ssh
```

**Крок 2.** Виходимо з VM та перевіряємо підключення без пароля:

```bash
ssh openclaw@192.168.56.10
```

Якщо потрапили без пароля — все налаштовано правильно.

#### Чому права такі важливі

SSH дуже прискіпливий до прав доступу на файли ключів. Якщо права виставлені неправильно — SSH **відмовиться** використовувати ключ і поверне помилку. Це захист: файл з правами `644` може прочитати будь-який користувач системи, а ключ — це секрет.

| Шлях                      | Права | Що це означає                       |
| ------------------------- | ----- | ----------------------------------- |
| `~/.ssh/`                 | `700` | Тільки власник може читати і писати |
| `~/.ssh/authorized_keys`  | `600` | Тільки власник може читати і писати |
| `~/.ssh/id_ed25519`       | `600` | Приватний ключ — тільки власник     |

#### Що таке `authorized_keys`

```text
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... your_email@example.com
```

Один рядок = один ключ. Можна додати кілька ключів (з різних машин) — по одному на рядок. Саме так ваш ключ і ключ GitHub Actions можуть одночасно мати доступ до одного сервера.

> 💡 **Швидкий спосіб скопіювати ключ без ручного copy-paste** — якщо є SSH-доступ з паролем (наприклад, на справжньому VPS):
>
> ```bash
> ssh-copy-id -i ~/.ssh/id_ed25519.pub openclaw@192.168.56.10
> ```
>
> `ssh-copy-id` спочатку підключиться з **паролем**, а потім автоматично виконає всі кроки.
>
> ⚠️ **На нашій VM це не спрацює.** Користувач `openclaw` створений як системний — у нього немає пароля взагалі (в `/etc/shadow` стоїть `!` — акаунт заблокований для парольного входу). Саме тому ми використовуємо bootstrap через `vagrant`: обходимо потребу в паролі, використовуючи вже існуючий привілейований канал. На реальному VPS провайдер зазвичай надає початковий пароль — і тоді `ssh-copy-id` є найшвидшим способом.

**Ключові поняття:**

- **`~/.ssh/authorized_keys`** — список публічних ключів, яким дозволено входити. Права: `600`.
- **`ssh-copy-id`** — утиліта-скорочення: сама підключається, виставляє права і додає ключ.
- **Bootstrap через довірений канал** — використання існуючого привілейованого з'єднання для налаштування нових облікових записів. Стандартна практика DevOps.

### 5. SSH Config: зручні псевдоніми

Після налаштування ключів команда підключення до `openclaw` виглядає так:

```bash
ssh openclaw@192.168.56.10
```

Коротко, але якщо є кілька серверів — набирати IP щоразу незручно. Файл `~/.ssh/config` дозволяє створити псевдонім для будь-якого підключення. Це не тільки зручно, але й є практикою реального DevOps:

```text
Host devvm-openclaw
    HostName     192.168.56.10
    User         openclaw
    IdentityFile ~/.ssh/id_ed25519
    Port         22
```

Тепер замість повної команди — просто:

```bash
ssh devvm-openclaw
```

SSH прочитає конфіг і підставить усі параметри самостійно.

> 💡 **Права на `~/.ssh/config`** мають бути `600` (читання/запис тільки власнику), інакше SSH може відмовитись використовувати файл.

**Ключові поняття:**

- **`~/.ssh/config`** — файл налаштувань SSH-клієнта. Дозволяє зберігати профілі підключення.
- **`Host`** — псевдонім, який ви будете використовувати в командному рядку.

### 6. SCP: передача файлів по SSH

Коли треба передати файл між вашим комп'ютером і сервером — використовується `scp` (Secure Copy Protocol). Він працює поверх SSH, тому ніяких додаткових налаштувань не потрібно — якщо SSH-підключення є, `scp` теж працює.

**Скопіювати файл з комп'ютера на сервер:**

```bash
scp ~/my_config.yaml vagrant@192.168.56.10:/home/vagrant/
# або, якщо є псевдонім у ~/.ssh/config:
scp ~/my_config.yaml devvm:/home/vagrant/
```

**Скопіювати файл з сервера на комп'ютер:**

```bash
scp vagrant@192.168.56.10:/var/log/app.log ~/Downloads/
```

**Скопіювати директорію (рекурсивно):**

```bash
scp -r ~/my_project/ vagrant@192.168.56.10:/opt/
```

> 💡 **SCP vs SFTP vs rsync:** SCP — найпростіший варіант для разових передач. `rsync` кращий для синхронізації великих директорій (передає тільки зміни). `sftp` — інтерактивний файловий менеджер поверх SSH. В нашому проєкті ми будемо переважно використовувати Git (Тема 5) для доставки коду на сервер, а `scp` — для разових конфігурацій.

### 7. SSH-тунелювання: доступ до прихованих сервісів

Це **найважливіший розділ теми** з погляду нашого проєкту.

У **Темі 3** ми залишили Python-сервер прив'язаним до `127.0.0.1:18789` — порт доступний лише зсередини VM, і з браузера на ноутбуці до нього не дістатись. Тоді це здавалось обмеженням. Тепер ми перетворимо це обмеження на перевагу: порт залишається закритим для всього Інтернету, але **ми** зможемо відкрити його у браузері через зашифрований SSH-тунель.

**Відповідь: SSH Port Forwarding (Local Tunnel).**

```bash
ssh -L 8080:localhost:18789 openclaw@192.168.56.10
```

Параметр `-L` має три частини через двокрапку: `[локальний_порт]:[хост_на_сервері]:[порт_на_сервері]`.

Розшифровуємо на нашому прикладі:

| Частина     | Значення                      | Простими словами                    |
| ----------- | ----------------------------- | ----------------------------------- |
| `8080`      | Порт на **вашому ноутбуці**   | Браузер йде на `localhost:8080`     |
| `localhost` | Адреса **з точки зору VM**    | `localhost` VM = `127.0.0.1` на VM |
| `18789`     | Порт **на VM**                | OpenClaw слухає тут                 |

> 💡 **Можна використати однаковий порт з обох сторін:**
>
> ```bash
> ssh -L 18789:localhost:18789 openclaw@192.168.56.10
> ```
>
> Тоді `http://localhost:18789/` у браузері відкриває OpenClaw на VM — рівно та сама адреса, що й коли сервіс запущений локально. URL не змінюється, тунель стає «прозорим». Саме так зазвичай налаштовують тунелі в реальній роботі.
>
> У нашому прикладі ми свідомо використовуємо `8080`, щоб наочно показати: цей порт — на **вашому** боці, а `18789` — на **сервері**.

```text
 [Ваш ноутбук]              [SSH-тунель]           [VM: localhost]
  localhost:8080                                      port :18789
        │                        │                        │
        │  1. Запит до :8080     │                        │
        │───────────────────────►│  2. Запит (зашифровано)│
        │                        │───────────────────────►│
        │                        │                        │ OpenClaw
        │                        │                        │ обробляє
        │                        │  3. Відповідь          │
        │  4. Відповідь          │◄───────────────────────│
        │◄───────────────────────│                        │
        │                        │                        │
```

Після запуску цієї команди — у браузері на вашому ноутбуці відкривається `http://localhost:8080` і ви бачите UI OpenClaw. Порт `18789` при цьому залишається невидимим для Інтернету.

> 🛠 **У лабораторній роботі** ми зробимо саме це: запустимо Python-сервер на `127.0.0.1:18789` (той самий, що в Темі 3) і відкриємо його у браузері через SSH-тунель. Ніяких змін у мережевих налаштуваннях VM — тільки один SSH-рядок.

Цю саму команду можна поєднати з псевдонімом та конфіг-файлом SSH:

```text
Host devvm-openclaw
    HostName     192.168.56.10
    User         openclaw
    IdentityFile ~/.ssh/id_ed25519
    Port         22
    LocalForward 8080 localhost:18789
```

Тепер `ssh devvm-openclaw` автоматично відкриє тунель при кожному підключенні.

> ⚠️ **Тунель живе тільки поки відкрите SSH-підключення.** Закрили термінал з SSH — тунель закрився. Це не проблема для розробки, але на Production для CI/CD використовуються інші механізми (Ansible сам відкриває тимчасове SSH-з'єднання).

**Різновиди форвардингу:**

| Тип                    | Прапор | Де відкривається порт | Для чого                                                                           |
| ------------------------- | ------------ | ---------------------------------------- | ----------------------------------------------------------------------------------------- |
| **Local Forward**   | `-L`       | На вашому ноутбуці       | Доступ до сервісів на сервері (наш випадок)            |
| **Remote Forward**  | `-R`       | На сервері                      | Дати серверу доступ до чогось на вашому ноутбуці |
| **Dynamic Forward** | `-D`       | SOCKS-проксі на ноутбуці | Проксі для браузера через сервер (як VPN)                   |

У нашому курсі використовуємо тільки Local Forward (`-L`).

**Ключові поняття:**

- **SSH Local Port Forwarding** — пробрасування (тунелювання) порту з серверу на локальну машину через SSH.
- **`-L local_port:remote_host:remote_port`** — формат параметра для Local Forwarding.
- **`LocalForward`** — директива в `~/.ssh/config` для автоматичного тунелю.

## ❓ Питання для обговорення на занятті

1. Ваш колега каже: «Навіщо морочитися з ключами? Я просто вимкну вимогу до пароля і буде зручніше». Що принципово не так з цим підходом?
2. Після `ssh-copy-id` ви підключаєтесь до VM без пароля. Потім переустановлюєте VM з нуля. Ви пробуєте підключитись — SSH видає `WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!`. Чому так сталося і що треба зробити?
3. Ви хочете дати доступ до VM ще двом колегам. Чи потрібно їм надсилати ваш приватний ключ? Як правильно організувати спільний доступ?
4. SSH-тунель запущений: `ssh -L 8080:localhost:18789 vagrant@192.168.56.10`. Ви відкриваєте `http://localhost:8080` у браузері — і бачите UI OpenClaw. Потім закриваєте термінал з SSH. Що станеться з браузерною вкладкою? Чому?
