## 🎯 Бізнес-задача

Ми підняли віртуальну машину — є «пісочниця», куди можна зайти. Та щойно ми заходимо всередину командою `vagrant ssh` — і одразу розгублені: де знаходяться системні налаштування? Де буде «жити» агент OpenClaw? Хто взагалі має право що робити на цьому сервері?

Більше того — є критична проблема безпеки: після `vagrant ssh` ми зайшли як користувач `vagrant` з повними правами `sudo`. Якщо ми запустимо агента від цього імені і хтось зламає агента — зловмисник отримає **повний контроль над сервером**. Нам потрібно створити окремого обмеженого користувача спеціально для агента.

**Зв'язок з проєктом:** У `My_AI_Assistant` агент OpenClaw запускається від імені системного користувача `openclaw` без прав `sudo`. Його реальні конфігурації зберігаються в `~/.openclaw/` — домашній теці цього користувача. У лабораторній роботі ми відпрацюємо принцип на прикладі `/opt`-директорії (згідно зі стандартом FHS — Filesystem Hierarchy Standard — для сервісного ПЗ), щоб розуміти цей підхід для будь-якого сервісу — а в Темі 6 автоматизуємо реальне налаштування через Ansible.

### 1. Файлова система Linux

На відміну від Windows (де є диск `C:\`, диск `D:\` тощо), Linux має **одне єдине дерево** з коренем `/` — і все, включно з «дисками», підключається (монтується) десь усередині цього дерева.

```text
/                        ← корінь (root) — єдина точка входу
├── home/                ← домашні теки користувачів
│   ├── vagrant/         ← ваша поточна домашня тека (/home/vagrant)
│   └── openclaw/        ← тека нашого агента (створимо сьогодні)
├── opt/                 ← додаткове ПО сторонніх розробників
│   └── openclaw/        ← файли агента (логіка, конфіги)
├── etc/                 ← конфігураційні файли системи та сервісів
├── var/                 ← змінні дані: логи, бази даних, кеш
│   └── log/             ← системні логи
├── tmp/                 ← тимчасові файли (очищуються при перезавантаженні)
├── usr/                 ← встановлені програми та бібліотеки
├── bin/ → usr/bin/      ← виконувані файли команд (ls, cp, cat...)
└── sys/, proc/          ← псевдо-файлова система (інформація про ядро та процеси)
```

**Аналогія з Windows:**

| Linux         | Windows                          | Призначення                  |
| ------------- | -------------------------------- | ---------------------------- |
| `/home/user/` | `C:\Users\User\`                 | Домашня тека користувача     |
| `/opt/`       | `C:\Program Files\`              | Встановлені програми         |
| `/etc/`       | Реєстр + `C:\Windows\System32\`  | Конфігурації системи         |
| `/var/log/`   | `C:\Windows\Logs\`               | Системні журнали             |
| `/tmp/`       | `C:\Windows\Temp\`               | Тимчасові файли              |

**Ключові поняття:**

- **`/` (root)** — корінь файлової системи, аналог «Цей комп'ютер» у Windows.
- **Абсолютний шлях** — починається з `/`, наприклад `/home/vagrant/project`.
- **Відносний шлях** — відносно поточної теки, наприклад `project/config`.
- **`~` (тільда)** — скорочення для домашньої теки поточного користувача (`/home/vagrant` або `/home/openclaw`).
- **`.` (крапка)** — поточна тека.
- **`..` (дві крапки)** — батьківська тека (на рівень вище).

### 2. Основні команди навігації та роботи з файлами

#### Навігація та перегляд

```bash
pwd                  # Print Working Directory — показати поточну теку
ls                   # List — список файлів у поточній теці
ls -l                # Детальний список (права, розмір, дата)
ls -la               # Детальний список + приховані файли (починаються з .)
ls -lh               # Детальний список з читабельними розмірами (KB, MB)
cd /opt              # Змінити теку (перейти в /opt)
cd ~                 # Перейти в домашню теку
cd ..                # Перейти на рівень вище
cd -                 # Повернутись у попередню теку
```

#### Перегляд вмісту файлів

```bash
cat /etc/hostname    # Вивести вміст файлу повністю
less /var/log/syslog # Перегляд великого файлу з прокруткою (q — вийти)
head -n 20 file.log  # Перші 20 рядків файлу
tail -n 20 file.log  # Останні 20 рядків файлу
tail -f file.log     # Слідкувати за файлом у реальному часі (Ctrl+C — вийти)
```

#### Робота з файлами та теками

```bash
mkdir my_folder             # Створити теку
mkdir -p /opt/openclaw/logs # Створити теку разом з батьківськими
touch config.txt            # Створити порожній файл
cp source.txt dest.txt      # Копіювати файл
cp -r folder/ new_folder/   # Копіювати теку рекурсивно
mv old.txt new.txt          # Перемістити або перейменувати
rm file.txt                 # Видалити файл
rm -r folder/               # Видалити теку рекурсивно (ОБЕРЕЖНО!)
```

#### Редагування тексту

```bash
nano config.txt   # Відкрити файл у простому редакторі nano
                  # Ctrl+O — зберегти, Ctrl+X — вийти
```

> 💡 **Порада:** `nano` — найпростіший редактор для початківців. `vim` та `emacs` значно потужніші, але мають круту криву навчання. На початку курсу використовуємо `nano`.

**Ключові поняття:**

- **`pwd`** — завжди показує, де ви зараз знаходитесь.
- **`ls -la`** — найбільш інформативний варіант `ls`; показує всі файли, права, власника.
- **`cat`** — швидко подивитись вміст невеликого файлу.
- **`tail -f`** — незамінний для моніторингу логів у реальному часі.
- **`nano`** — зручний текстовий редактор для початківців.

### 3. Права доступу та власники

Це одна з найважливіших тем для розуміння безпеки сервера. У Linux кожен файл і тека мають **власника**, **групу** та набір **прав доступу**.

#### Як читати вивід `ls -la`

```text
drwxr-xr-x  2 openclaw openclaw 4096 Mar 12 20:00 /opt/openclaw
│└──┬───┘    │ └──┬───┘ └──┬───┘
│   │        │   │         │
│   │        │  власник  група
│   │       кількість посилань
│   └── права (3 групи по 3 символи: owner | group | others)
└── тип: d=тека, -=файл, l=символічне посилання
```

#### Три групи прав: власник | група | інші

```text
rwxr-xr--
│││└──┬──┘└─┬─┘
│││   │      └── інші (others): r-- = тільки читання
│││   └── група (group): r-x = читання + виконання
│││
│││ r = read    (читання)
││└ x = execute (виконання — для файлів: запустити; для тек: увійти)
│└─ w = write   (запис)
└── r (перший символ — власник)
```

#### Зміна прав: `chmod`

```bash
# Символьний запис (зрозуміліший для початківців)
chmod u+x script.sh    # u=user/owner: додати право виконання
chmod g-w file.txt     # g=group: зняти право запису
chmod o-rwx secret.txt # o=others: зняти всі права

# Числовий запис (швидший, широко використовується)
# r=4, w=2, x=1 — складаємо для кожної групи
chmod 755 /opt/openclaw   # rwxr-xr-x (власник: все; решта: читання+виконання)
chmod 644 config.txt      # rw-r--r-- (власник: читання+запис; решта: тільки читання)
chmod 600 secret.key      # rw------- (тільки власник може читати та писати)
```

#### Зміна власника: `chown`

```bash
chown openclaw /opt/openclaw           # змінити власника
chown openclaw:openclaw /opt/openclaw  # змінити власника та групу
chown -R openclaw:openclaw /opt/openclaw/ # рекурсивно (для всього вмісту теки)
```

> 🔑 **Принцип найменших привілеїв (Principle of Least Privilege):** кожен процес (і користувач) повинен мати лише ті права, які необхідні для виконання конкретного завдання — і нічого більше. Саме тому агент `openclaw` не матиме `sudo`.

**Ключові поняття:**

- **`chmod`** — змінює права доступу до файлу або теки.
- **`chown`** — змінює власника файлу або теки.
- **Числовий запис `chmod`** — r=4, w=2, x=1; `755` = rwxr-xr-x.
- **Принцип найменших привілеїв** — не давай більше прав, ніж потрібно.

### 4. Користувачі, групи та sudo

#### Хто ти є?

```bash
whoami          # Вивести ім'я поточного користувача
id              # Детальна інформація: uid, gid, групи
id openclaw     # Інформація про конкретного користувача
```

#### Типи користувачів

У Linux є три категорії:

| Тип        | UID              | Опис                                             |
| ---------- | ---------------- | ------------------------------------------------ |
| `root`     | 0                | Суперкористувач, необмежені права                |
| Системні   | 1–999            | Сервіси (nginx, postgres, openclaw)              |
| Звичайні   | 1000+            | Реальні люди (vagrant, maxim, student)           |

#### Що таке sudo?

`sudo` (Superuser Do) — дозволяє звичайному користувачеві виконати одну конкретну команду з правами `root`. Це як «тимчасово позичити ключ від всього будинку» — тільки для одного конкретного діла.

```bash
sudo apt-get update    # виконати apt-get update з правами root
sudo cat /etc/shadow   # прочитати файл паролів (звичайно заборонено)
sudo -l                # переглянути, які команди дозволено виконувати з sudo
```

> ⚠️ **Чому агент не повинен мати sudo?** Якщо зловмисник зламає процес агента — він отримає права того користувача, від якого агент запущений. Якщо це `root` або користувач з `sudo` — зловмисник отримає повний контроль над сервером. Якщо це `openclaw` без `sudo` — він заблокований у домашній теці агента.

**Реальна продакшн-практика:** На справжньому VPS зазвичай існують два окремих облікових записи з різними ролями:

| Користувач | Shell  | SSH-ключ | sudo | Призначення                                         |
| ---------- | ------ | -------- | ---- | --------------------------------------------------- |
| `openclaw` | `bash` | ✅       | ❌   | Запускає агента, скрипти оновлення, файли агента    |
| `deploy`   | `bash` | ✅       | ✅   | Адміністрування сервера, встановлення пакетів       |

Зверніть увагу: `openclaw` **може** мати SSH-доступ (це зручно — наприклад, для запуску скриптів оновлення або роботи з файлами агента через VS Code Remote SSH). Але у нього немає `sudo`, тому він не може вийти за межі своїх файлів та зламати систему. **Ключова межа безпеки — не SSH, а відсутність `sudo`.**

Варіант з `nologin` (що ми практикуємо в лабораторній роботі) — ще жорсткіший підхід для сервісів, які взагалі не потребують інтерактивного управління. Обидва підходи правильні — вибір залежить від операційних потреб. Саме такий поділ ролей ми побачимо в **Темі 4 (SSH)** та **Темі 6 (Ansible)**.

#### Керування користувачами: підготовка оточення для агента

Уявімо ситуацію: агент OpenClaw встановлений і готовий до запуску. Але від якого імені він буде працювати? Якщо запустити його від `root` — це катастрофа з точки зору безпеки. Якщо від `vagrant` — цей користувач є тільки у нашій тестовій VM. Нам потрібен **власний обліковий запис спеціально для агента**.

##### Крок 1. Створити користувача

```bash
sudo useradd --system --no-create-home --shell /usr/sbin/nologin openclaw
```

Що означають прапори:

- `--system` — створює системного користувача з UID у діапазоні 1–999 (не людський акаунт). Це сигнал операційній системі, що цей обліковий запис — для сервісу, а не для людини.
- `--no-create-home` — не створювати `/home/openclaw`. У суворому випадку робоча директорія агента знаходиться в `/opt/openclaw` — і ми створимо її самі з правильними правами. Альтернативна практика (описана далі) — дати домашню теку і зберігати конфіг в `~/.openclaw/`.
- `--shell /usr/sbin/nologin` — замість bash встановлюємо «фальшивий» shell, який при спробі входу одразу завершується з повідомленням: «This account is currently not available.» Ніхто, навіть адміністратор, не зможе увійти в систему напряму від імені цього користувача — можна лише запустити процес з правами `openclaw` через `sudo -u openclaw`.

> 💡 **Альтернатива:** Як ми вже обговорювали, якщо потрібен інтерактивний доступ до файлів (наприклад, для скриптів оновлення або VS Code Remote SSH), дають `bash` замість `nologin`. Вибір залежить від операційних потреб.

##### Крок 2. Підготувати робочу директорію

Наступне питання: де зберігатимуться файли агента? Чому `/opt/openclaw`, а не `/home/openclaw`?

У Linux склалася чітка конвенція (стандарт FHS — Filesystem Hierarchy Standard):

- `/home/` — для людей. Тут живуть особисті файли, налаштування, документи реальних користувачів.
- `/opt/` — для додаткового програмного забезпечення сторонніх розробників. Сервісний процес — не людина, він живе в `/opt`.

```bash
sudo mkdir -p /opt/openclaw
```

Прапор `-p` означає «створити також усі відсутні батьківські директорії». Якщо б `/opt` не існувало — він створив би і його. Безпечна і зручна звичка.

> 🏭 **Як це виглядає в реальному OpenClaw:** Сам OpenClaw (TypeScript/Node.js-застосунок) зберігає конфігурацію у `~/.openclaw/` — тобто в домашній теці свого користувача. Тому в продакшн-налаштуванні користувач `openclaw` **має** домашню теку (`/home/openclaw`). Ми практикуємо `/opt`-модель, бо вона відповідає FHS (Filesystem Hierarchy Standard) — міжнародному стандарту розташування файлів в Linux — і ці знання застосовні до будь-якого сервісу (nginx, postgresql, власний Python-скрипт). Обидва підходи законні: вибір залежить від того, як влаштований конкретний застосунок.

##### Крок 3. Передати директорію у власність агента

Коли ми виконали `mkdir` з `sudo`, директорію було створено від імені `root`. Тому зараз:

```bash
ls -la /opt/
# drwxr-xr-x 2 root root 4096 ...  openclaw
```

Директорія належить `root`. Це означає: процес `openclaw` взагалі не зможе записати жодного файлу у свою ж робочу директорію — ні логів, ні конфігурацій. Він запуститься і одразу впаде з помилкою `Permission denied`.

Щоб виправити це:

```bash
sudo chown openclaw:openclaw /opt/openclaw
```

Тепер директорія належить саме `openclaw`. Команда `chown власник:група шлях` змінює і власника, і групу одночасно. Після цього:

```bash
ls -la /opt/
# drwxr-xr-x 2 openclaw openclaw 4096 ...  openclaw
```

Процес агента зможе читати та записувати у свій каталог — і тільки в нього.

##### Перевірка результату

```bash
id openclaw
ls -la /opt/
```

#### Переключення між користувачами

Іноді потрібно виконати команди від імені іншого користувача — наприклад, протестувати, що бачить `openclaw` у системі:

```bash
sudo -u openclaw bash   # Запустити bash від імені openclaw (якщо ваш поточний user має sudo)
whoami                  # openclaw
exit                    # Повернутись до попереднього користувача
```

> 📌 `su - openclaw` в нашому випадку не спрацює, бо shell встановлено в `nologin`. Команда `sudo -u` обходить це обмеження — вона запускає процес від імені `openclaw`, але не через його shell.

**Ключові поняття:**

- **`root`** — суперкористувач Linux; UID=0; може все.
- **`sudo`** — механізм делегування прав; виконати команду як root.
- **Системний користувач** — без домашньої теки та без права на інтерактивний вхід; використовується для сервісів.
- **`useradd --system`** — створити системного (не людського) користувача.
- **`/usr/sbin/nologin`** — «фальшивий» shell, що забороняє інтерактивний вхід.

### 5. Моніторинг системи: RAM, CPU, диск, процеси

Адміністратор сервера постійно стежить за станом системи. Ось основні інструменти.

#### Оперативна пам'ять та SWAP

```bash
free -h
```

Приклад виводу:

```text
               total        used        free      shared  buff/cache   available
Mem:           1.9Gi       512Mi       800Mi        10Mi       600Mi       1.3Gi
Swap:          1.0Gi         0B       1.0Gi
```

| Поле        | Значення                                                          |
| ----------- | ----------------------------------------------------------------- |
| `total`     | Скільки RAM встановлено (у VM — 2 GB)                             |
| `used`      | Скільки зайнято зараз                                             |
| `available` | Скільки реально доступно (враховує кеш)                           |
| **Swap**    | Файл підкачки на диску — використовується, коли не вистачає RAM   |

> 💡 **Swap** — це «аварійна пам'ять» на диску. Якщо `Swap used` > 0, то системі не вистачає RAM — час або збільшити RAM, або оптимізувати застосунки.

#### Завантаження процесора та процеси

```bash
top        # Інтерактивний моніторинг (q — вийти, k — kill процес)
htop       # Покращена версія top (якщо встановлена: sudo apt install htop)
ps aux     # Список всіх процесів у системі (статичний знімок)
ps aux | grep openclaw  # Знайти процеси агента
```

Розшифровка заголовка `top`:

```text
top - 20:00:00 up 2:30,  1 user,  load average: 0.05, 0.10, 0.08
                          └─────┘  └────────────────────────────────┘
                          сесій    завантаження за 1, 5, 15 хвилин
```

> 💡 `load average` показує **три числа через кому**: середнє навантаження за останні **1 хвилину**, **5 хвилин** і **15 хвилин**. Значення `1.0` на однопроцесорній машині означає 100% завантаження (черга процесів заповнена). На нашій VM (2 ядра) нормально — до `2.0` за будь-який з трьох проміжків.
>
> Як читати три числа: якщо `1 хв` значно вище за `15 хв` — навантаження щойно зросло (сплеск). Якщо навпаки `15 хв` вище — система поступово розвантажується після піку.

### 6. Змінні оточення (Environment Variables)

Змінні оточення — це пари «ключ=значення», які доступні будь-якому процесу в системі. Це стандартний спосіб передавати конфігурацію застосунку без вшивання її в код.

```bash
echo $HOME          # Домашня тека поточного користувача
echo $USER          # Ім'я поточного користувача
echo $PATH          # Де система шукає виконувані програми
echo $SHELL         # Поточний shell (зазвичай /bin/bash)
env                 # Показати всі змінні оточення
```

#### Встановити змінну

```bash
# Тимчасово (тільки для поточної сесії)
export MY_VAR="hello"
echo $MY_VAR

# Постійно — додати до ~/.bashrc
echo 'export MY_VAR="hello"' >> ~/.bashrc
source ~/.bashrc    # Застосувати зміни без перезапуску
```

#### Що таке `.bashrc`?

Файл `~/.bashrc` — скрипт, який автоматично виконується кожного разу, коли ви відкриваєте новий термінал (нову bash-сесію). Саме тут зберігаються постійні змінні оточення, псевдоніми команд та налаштування bash.

```bash
cat ~/.bashrc      # Переглянути поточний bashrc
```

> 🔑 **Чому це важливо для нашого агента?** Пізніше в курсі API-ключі та токени агента зберігатимуться як змінні оточення — а НЕ вшиватимуться в код. Так ми уникаємо випадкового публікування секретів у Git.

#### Команда `source` — застосувати зміни без перезапуску сесії

Коли ви редагуєте `~/.bashrc` (або будь-який інший конфігураційний файл shell), зміни **не набувають чинності автоматично** — bash вже запущений і не знає, що файл змінився. Щоб зміни спрацювали без того, щоб закривати і відкривати термінал заново, використовують `source`:

```bash
# Крок 1: Додати нову змінну до .bashrc
echo 'export TELEGRAM_TOKEN="1234567:ABCdef"' >> ~/.bashrc

# Крок 2: Перевірити — ще не спрацювало
echo $TELEGRAM_TOKEN
# (порожній рядок — змінна ще не існує в поточній сесії)

# Крок 3: Застосувати зміни
source ~/.bashrc

# Крок 4: Тепер змінна доступна
echo $TELEGRAM_TOKEN
# 1234567:ABCdef
```

> 💡 `source файл` і `. файл` — це одне й те саме. Крапка (`.`) — скорочений синтаксис POSIX. В bash обидва варіанти рівнозначні.

**Ключові поняття:**

- **Змінна оточення** — пара ключ=значення, доступна всім процесам.
- **`export`** — зробити змінну доступною для дочірніх процесів.
- **`$PATH`** — список директорій, де bash шукає команди.
- **`~/.bashrc`** — файл налаштувань bash, що виконується при вході.
- **`source`** — негайно застосувати зміни файлу без перезапуску.
